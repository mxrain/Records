# Records 项目重构计划

## 总结

逐步将项目从 "GitHub JSON 作为 CMS + 双重状态管理 + 混用 UI 库" 的架构，重构为 "PostgreSQL 主存储 + RTK Query 统一状态 + shadcn/ui 统一 UI" 的现代化架构。

采用**五阶段逐步迁移**，每阶段独立可运行，不影响生产发布。

---

## 当前状态分析

### 现有问题清单

| 问题 | 严重度 | 说明 |
|------|--------|------|
| 双重状态管理 | 高 | `createAsyncThunk` (resourcesSlice/listSlice/categoriesSlice) 与 RTK Query (resourcesApi/listApi) 并存，两套代码做同一件事 |
| antd 残留 | 低 | 仅 `HeatmapCard.tsx` 使用 `Spin` + `Popover`，但整体引入了 2MB+ 的 antd |
| GitHub 作为 CMS | 中 | 所有数据走 GitHub REST API，网络延迟高、有 API 频率限制、无事务支持、JSON 文件不适合结构化查询 |
| middleware.ts 废弃 | 低 | Next.js 16 废弃 middleware，需迁移到 proxy |
| 数据库层闲置 | 中 | `lib/db.ts` PostgreSQL 连接层已写好，但仅用于初始化示例，未被 API 层使用 |
| 文件/组件组织混乱 | 中 | `app/components/` 和 `components/ui/` 两套 shadcn 组件并存；`app/(main)/components/` 和 `app/components/` 职责不清 |
| CSS Modules + Tailwind 混用 | 中 | 部分组件用 `*.module.css`，部分用 Tailwind，维护心智负担 |
| 无测试 | 高 | 项目零测试覆盖，重构/升级依赖后只能靠 Vercel 构建来验证 |
| 废弃 npm 依赖 | 低 | `npmlog`、`gauge`、`rimraf@3`、`inflight`、`glob@7` 在构建时报警告，来自间接依赖 |

### 现有资产（保留）

- Redux store 结构设计合理（tabs/resources/list/categories/changeRecords）
- RTK Query API 层已经搭建完成（resourcesApi/listApi/categoriesApi/tagsApi/iconsApi）
- shadcn/ui 40+ 组件基础扎实（components/ui/）
- JWT + Turnstile 认证体系完整可用
- 设计令牌系统（design-tokens.css + tailwind.config.ts）设计良好
- `lib/api.ts` 的 `syncWithGithub` 保持为备份同步
- changelog/变更记录系统可作为数据双写的回放基础

## 第一阶段：基础设施迁移（不影响前端）

### 目标
将数据库从"本地声明"切换为"免费云服务"，为后续数据层迁移做好准备。

### 具体变更

**1.1 数据库配置切换**
- 文件：`lib/db.ts`、`.env.template`、`docker-compose.yml`
- 变更：
  - 新增 `DATABASE_URL`（Neon）、`REDIS_URL`（Upstash）、`R2_*` 环境变量
  - 修改 `lib/db.ts` 支持 `DATABASE_URL` 连接串（目前用的是 pg.Pool 参数拆解）
  - 新增 `lib/storage/r2.ts`：Cloudflare R2 对象存储客户端（替代 MinIO for 图片/文件）
  - 新增 `lib/cache/redis.ts`：Upstash Redis 客户端（替代现有 `redis` 包配置）
  - 更新 `.env.template` 列出所有新环境变量
  - 更新 `docker-compose.yml` 为注释引用（远程服务不再本地起）

**1.2 数据库 Schema 对齐**
- 文件：`lib/db-init.ts`
- 变更：
  - 新增 `resources` 表的完整字段（与当前 GitHub JSON 中字段对齐：json_data JSONB、sync_status、github_sha 等）
  - 新增变更日志表 `change_logs`（用于 GitHub 备份同步）
  - 新增索引优化（GIN 索引用于 JSONB 查询）
  - 在 Neon 上执行 Schema 初始化

### 验收标准
- `npm run db:init` 在 Neon 上成功创建所有表
- Redis 连接可读写
- R2 bucket 可上传/下载文件

## 第二阶段：数据层迁移（GitHub → PostgreSQL）

### 目标
将资源数据的读写从 GitHub REST API 迁移到 PostgreSQL/Redis/R2，GitHub 仅作为异步备份。

### 具体变更

**2.1 核心数据访问层新建**
- 新增文件：`lib/data/resources.ts` — 资源 CRUD（直接操作 PostgreSQL）
- 新增文件：`lib/data/categories.ts` — 分类 CRUD
- 新增文件：`lib/data/list.ts` — 列表配置 CRUD
- 新增文件：`lib/data/tags.ts` — 标签 CRUD
- 新增文件：`lib/data/media.ts` — 图片上传到 R2 + 记录到 PostgreSQL

**2.2 Redis 缓存层**
- 新增文件：`lib/cache/cacheUtils.ts` — 整合现有 `lib/cacheUtils.ts`，改用 Redis 替代内存 Map
- 模式：查询先查 Redis → 未命中查 PostgreSQL → 写入 Redis（TTL 5分钟）

**2.3 API 路由重构**
- 文件：`app/api/resources/route.ts`、`app/api/resources/[id]/route.ts`
- 变更：
  - GET 方法：先查 Redis 缓存 → PostgreSQL
  - POST/PATCH/DELETE：写 PostgreSQL → 失效 Redis 缓存 → 写入 change_logs 表 → 返回响应
- 文件：`app/api/categories/route.ts`、`app/api/tags/route.ts`、`app/api/list/route.ts`
- 变更：同上，从 GitHub 读取改为 PostgreSQL 读取

**2.4 GitHub 异步备份层**
- 新增文件：`lib/sync/github-backup.ts`
- 逻辑：监听 `change_logs` 表 → 定期/批量将变更写入 GitHub JSON 文件
- 保留：`lib/api.ts` 中 `syncWithGithub` 的核心逻辑，改造为备份专用
- 变更记录系统：保持 `changeRecordsSlice` 的前端交互，后端改为写 `change_logs` 表

### 验收标准
- API 层 100% 从 PostgreSQL 读写
- Redis 缓存命中率可通过日志观察
- GitHub 备份延迟 < 60 秒
- 前端所有页面功能正常（无感知迁移）

## 第三阶段：状态管理统一（createAsyncThunk → RTK Query）

### 目标
删除冗余的 `createSlice` + `createAsyncThunk` 方案，统一使用 RTK Query。

### 具体变更

**3.1 删除废弃 Slice**
- 文件变更：
  - `app/store/features/resources/resourcesSlice.ts`
    - 删除 `fetchResources` async thunk
    - 保留 UI 状态（selectedResources、filters 等）
    - 所有数据获取改用 `resourcesApi`
  - `app/store/features/categories/categoriesSlice.ts`
    - 删除 `fetchCategories` async thunk
    - 保留 UI 状态（expandedKeys 等）
    - 数据获取改用 `categoriesApi`
  - `app/store/features/list/listSlice.ts`
    - 删除 `fetchList` async thunk
    - 数据获取改用 `listApi`

**3.2 增强 RTK Query API**
- 文件：`app/store/api/resourcesApi.ts`、`app/store/api/listApi.ts` 等
- 变更：
  - 确保所有 endpoint 覆盖旧 async thunk 的功能
  - 添加 `providesTags` / `invalidatesTags` 实现自动缓存失效
  - 添加乐观更新（optimistic update）用于增删改操作

**3.3 统一数据流模式**
- 模式：`Component → useQuery/useMutation (RTK Query) → API Route → Data Layer → PostgreSQL/Redis`
- 删除 `lib/api.ts` 中的 deprecated 函数（`fetchCategories`、`fetchResources` 等）

### 验收标准
- `grep -r "createAsyncThunk" app/store/` 仅出现在 changeRecordsSlice（保留）
- 所有数据获取使用 RTK Query hooks
- 前端功能无回归

- 删除 `app/components/ui/` 目录（已无内容）

## 假定与决策

| 决策 | 理由 |
|------|------|
| 不处理 category-redesign/homepage-redesign 设计稿 | 用户明确要求暂不处理 |
| 保留自定义 JWT 认证 | 现有方案成熟稳定，无需引入外部服务 |
| 使用 Neon/Upstash/R2 免费层 | 额度足够项目使用（0.5GB PG + 256MB Redis + 10GB R2） |
| 保留 GitHub 备份 | 历史数据安全 + 可在 GitHub 上直接查看 JSON 原始数据 |
| 逐步迁移（非大爆炸） | 每阶段独立验证，降低风险 |
| 使用 Vitest 而非 Jest | 与项目现有的 tsconfig/Turbopack 生态更兼容 |
